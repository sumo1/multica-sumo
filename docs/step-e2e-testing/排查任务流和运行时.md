# 排查任务流和运行时

读这个文件的场景：任务 queued 不动、规划卡住、Claude Code 403、桌面端 ④ 流空、下游 agent 没拿到上游产出、或者 UI 不知道哪个 agent/model 在跑。

## 先定位层级

| 症状 | 先查 |
|---|---|
| `queued` 不认领 | daemon 是否在线、task wakeup 是否丢失、重启 daemon 是否 sweep |
| planning 卡住或 prompt 不对 | daemon 二进制是否旧、server/daemon 是否都重建 |
| Claude Code `403` / `Not logged in` | daemon env、proxy、`CLAUDE_CODE_*` 是否被过滤、agent `custom_env` |
| `curl localhost` 502/403 | 是否忘了 `--noproxy '*'` |
| daemon token 401 | server 是否连错 DB，真实数据在 `multica` |
| ④ 流空但 DB 有 message | `ReportTaskMessages` 是否用共享 workspace resolver 广播 |
| 下游 agent 重复推导上游产出 | execute 节点 context 是否带 `upstream_output` |
| 不知道谁跑的 | 看 task/subtask 级 agent/runtime/model 归属，不按每条 message 猜 |

## 403 / Not logged in

高概率不是账号真没登录，而是 daemon 子进程环境错。

检查顺序：

1. daemon 是否用 `env -i` 启动。用了就是错。
2. 是否只 unset 代理变量，而不是清空全部 env。
3. shell 里 `claude -p "PONG"` 能跑，但 daemon 子进程不能跑时，差异一定在 daemon env 处理。
4. `CLAUDE_CODE_USE_BEDROCK` 会被 daemon 过滤，必须通过 agent `custom_env` 重新注入。
5. 同步 agent 的 `mcp_config` 应为 NULL；`runtime_mode` 应与 runtime 一致。

## ④ 流空

不要只看 `task_messages` 是否落库。实时 UI 依赖 WS `task:message`。

常见根因：handler 自己用 issue/chat/autopilot FK 推导 workspace，遇到 goal FK-less 任务返回空，broadcast 被跳过。

修法：所有 task workspace 解析统一走共享 resolver，并覆盖所有 context type。

## 上游产出丢失

execute 节点也要拿 depends_on 上游 result。verify 节点拿上游 spec+result 是审查需求，execute 节点至少要拿上游 result 继续工作。

memory 不是任务间传输介质。不要让下游 agent 去 memory 或设计文档里重新推导上游产物。

## model / runtime / agent 归属

归属按任务或子任务显示，不按每条 message 显示。

当前策略：

- 配置值先填；
- `task_usage` 的实际 model 可覆盖配置值；
- goal 主记录和 subtask 响应都要带 agent/runtime/model；
- 前端状态树、子任务流、总控头部渲染这些字段。
