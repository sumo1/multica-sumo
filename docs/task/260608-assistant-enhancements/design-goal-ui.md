## 一、设计参考调研（Codex / Claude Code）

### Claude Code「agent teams」——几乎就是我们的 PMO 模型

| 概念 | Claude Code | 对应我们的 |
|------|------------|-----------|
| Team lead | 创建团队、拆任务、派发、汇总 | **PMO（= squad leader）** |
| Teammates | 各自独立上下文执行被分配任务 | **执行角色 agent（= squad member）** |
| **Shared task list** | 工作项列表，状态 pending → in_progress → completed，**任务间可声明依赖**，依赖未完成则不可 claim | **goal 的子任务 DAG** |
| Mailbox | agent 间消息 | 子任务/角色间协调（后期） |

关键印证：**任务列表 + 依赖 + 三态**正是我们 design.md 里定的 goal_run 子任务模型。Claude Code 已验证这套抽象可行。

### Claude Code「agent view」——多 agent 监控界面的视觉范式

终端里一屏管所有后台 session，**按状态分组**：

```
Pinned        ✽ clawd walk cycle     Write assets/sprites/...        3m
Ready for review ∙ jump physics      Opened PR with collision fix    2h
Needs input   ✻ power-up design      needs input: double jump?       1m
Working       ✽ collision detection  Edit src/.../CollisionSystem    2m
Completed     ✻ title screen         result: menu/options done       9m
```

每行 = 一个 agent/任务：**状态图标（色+动画）+ 名称 + 当前动作 + 时长**。状态：Working(动画) / Needs input(黄) / Idle(暗) / Completed(绿) / Failed(红) / Stopped(灰)。

→ 这正是我们「状态树/PMO 树」每个节点该长的样子：**图标 + 角色名 + 当前在做什么 + 耗时**。

### Codex 右上角 Progress 面板（之前你提供的截图）

- **Environment 区**：分支、改动量(+/-)、commit/push/review 操作
- **Progress 区**：拆解步骤清单，每条带状态圈（✓ 完成 / ◌ 进行中 / ○ 待办）
- **Sources 区**：引用/产物

→ 对应我们第四栏的「整体任务拆解步骤 + 当前进展」。

## 二、与 multica 现有 squad 的关系（关键复用判断）

调研结论：**squad 已是 PMO 雏形，不新造「团队」概念。**

| squad 已有 | 对应 PMO 需要 |
|-----------|--------------|
| `squad.leader_id`(agent) | PMO 队长 ✅ |
| `squad_member`(agent/member + role) | 执行角色成员 ✅ |
| `squad.instructions` | 团队级指令 ✅ |
| leader 接 issue → 评估 → @mention 委派（squad_briefing.go） | PMO 派发雏形 ✅ |
| `ListSquadMemberStatus`（成员 working/idle/offline + 当前 issue） | 成员实时状态 ✅ |
| squads-page / squad-detail-page | 列表/详情页 ✅ |

**squad 缺的（= 我们要建的）：**
1. **Goal 实体**：squad 层面的整体目标 + 拆解（现在只有零散 issue）
2. **子任务 DAG**：任务间依赖声明（现在只有 issue 父子）
3. **批量/并行派发编排**：现在 leader 一次只 @mention 一个
4. **群聊 chat**：chat_session 现在只绑单 agent，不能跟 squad 对话
5. **状态可视化**：leader 评估历史、goal 进度树前端

> 结论：**「团队」= 扩展 squad**。第二栏的「团队」列表 = squad 列表。

### 各栏职责

**① 导航栏**（现有，仅新增「团队/小队」入口）
- 复用现有 app-sidebar。「团队」= 现有「小队」菜单项（或重命名）。

**② 团队列表**（= squad 列表）
- 顶部「+ 新建团队」。
- 每项：团队名 + PMO（leader）名 + 成员数 + 整体状态点。
- 点击 → ③④ 切到该团队。

**③ 聊天对话（主体）**
- **顶部条**：团队名 · 模式开关（普通聊天 / goal 模式）· **右上角整体进展（第 N/M 步 + 进度条）**。
- **对话流**：用户 ↔ PMO 的主对话；goal 模式下，**内嵌「子进程卡片」**（借鉴 Claude Code agent view 的行：图标+角色+当前动作），实时显示每个执行角色在干什么。
- **输入框**：给团队下达目标（goal）或普通消息。

**④ 状态区（Codex Progress 式，三段）**
- **整体进度**：Goal 标题 + 进度条 + 第 N/M 步 + 总耗时。
- **PMO 角色树**（你定的 goal → 子任务 → 角色）：
  ```
  ◆ Goal
  ├ ✓ 子任务1  └ ◆角色 done
  ├ ◌ 子任务2  └ ◆角色 运行中 + 当前动作
  └ ○ 子任务3  └ ○角色 待办 (依赖 1,2)
  ```
  节点状态图标对齐 Claude Code agent view：✓完成/◌进行中(动画)/○待办/✗失败。
- **来源/产物**：PR、trace、生成文件链接。

### 三栏为什么不够 → 四栏的依据
- 中间栏若塞「团队创建 + 任务状态 + PMO 树」会过载（你的原始担忧成立）。
- 拆成：②管「选哪个团队」、③管「对话+实时子进程」、④管「结构化状态(树+进度)」。
- 团队**创建**走弹框/独立页（不常驻占栏），不挤进 ②。

## 四、普通聊天 vs goal 模式（同一对话入口，承接需求一）

- 第三栏顶部有**模式开关**。
- **普通聊天**：跟单个 agent（或 PMO）一对一，④状态区收起/只显示 runtime。
- **goal 模式**：触发 PMO 拆解 → ④状态区展开 PMO 树 + 进度，③对话流内嵌子进程卡片。

---

## 五、设计决策（已定，2026-06-09）

1. **团队 = 复用 squad，不重命名，分层。**
   squad 保持原样作为底层数据模型（已有 issue 指派、leader 委派、成员管理、现成页面，重命名会破坏现有功能/i18n/reserved-slug）。「团队」是 squad 的**一个新视图/能力**：同一个 squad，既能在「小队」页配人，也能在「助理」里以「团队」身份接 goal。**底层一套 squad，上层两个入口**，零新表、零破坏。

2. **第四栏按需：goal 模式展开，普通聊天收起。**
   普通聊天（需求一，单 agent）没有子任务树/进度，第四栏会空着浪费。切到 goal 模式 → 第四栏滑入；切回普通 → 滑出。第四栏收起时，③顶部保留轻量「第 N/M 步」进度条，完整树留给④。

3. **不双视图：③=时间线，④=结构快照，互补不重复。**
   - ③对话流 = 时间维度（PMO 说了什么 + 子进程关键消息流，按时间滚动）。
   - ④树 = 空间维度（goal→子任务→角色层级 + 每节点实时状态快照）。
   - 同一信息不渲染两遍。看「实时发生什么」→③；看「整体结构/进度」→④。

4. **新建团队 = 弹框，复用现有 squad 创建流程。**
   助理第二栏「+ 新建团队」直接复用 squads-page 已有的创建弹框/逻辑，不另造、不跳独立页（避免丢当前对话上下文）。

5. **点 PMO 树节点 → ③对话流过滤到该角色子流，不切走。**
   点角色节点 → ③高亮/过滤出该角色子进程消息；点 Goal 根节点 → ③回到全局混合时间线。保持 PMO 全局视角（不切进孤立子对话）。心智同 Claude Code agent-view 的 "peek"，但用过滤而非弹层。进阶（后期）：节点给次级入口「展开完整子对话」。

**共同逻辑**：最大复用 squad、信息不重复渲染、一次性操作走弹框、保持 PMO 全局视角。

## 五点五、Goal 模式交互逻辑（端到端，已对齐 2026-06-09）

### 前提：goal 模式先有团队（squad）
对话对象是团队（squad），不是单 agent。PMO = squad leader；执行角色 = squad member。没有就先在②点「+ 新建团队」（复用 squad 创建弹框）。

### 三阶段（讨论 → 确认 → 执行）

**阶段一：讨论 / 设计（你 ↔ PMO 多轮）**
- 用户在③下达粗目标 → PMO 反问澄清（验证码走短信还是邮件？旧登录保留兼容吗？）→ 多轮收敛。
- 这一阶段④显示"草案/拟拆解"，不是执行状态，不派任何 agent 干活。
- **本质 = harness 的 task-designer + 双契约形成过程**：讨论产出一份需求文档。

**需求文档进 git（核心，呼应双 SSOT）：**
- PMO 讨论产出 → 写成真实 markdown：**目标工程 repo 的 `docs/task/{goal-id}/requirement.md`**（决策：进**目标对应的工程 repo**，文档与代码同仓同审，harness 灵魂）。
- 后续 `design.md`、`memory/` 同样落该 repo。
- **写入方向说明**：这是"平台讨论 → 写进 repo"（平台是本次产出的源，产出后以 repo 为 SSOT）。与之前定的"已有沉淀 单向 repo→平台"不冲突——分工：repo 已有的读上来，平台新产生的写下去。

**阶段二：确认（一道闸门）**
- PMO 给出最终任务设计（= 需求文档定稿 + 子任务 DAG）→ 用户点**「确认执行」**。
- 不点确认不派 agent。对应 Claude Code plan 审批 / harness"契约确认后才施工"。避免拆错直接烧 token。

**阶段三：执行（并行/串行）**
- PMO 按 DAG 调度：无依赖子任务并行下发，有依赖的等前置完成（依赖完成自动解锁）。
- 执行复用 chat-task 路（见六）。

### 子任务可见性
④树点开任意子任务节点可见：**目标(spec) + 角色 + runtime + 状态 + 依赖 + 当前动作 + 产出/diff**。点节点→③过滤到该角色子流。

### 失败处理 + 用户干预（harness「打回循环」落地，已定）

1. **默认行为**：失败 → PMO **自动重试 1-2 轮**（带失败原因当上下文）→ 仍失败 → **升级到用户裁决**（doctrine 的 N 轮升级，防越修越乱）。
2. **彻底失败/跳过**：**继续其余 + 阻塞下游**——能跑的子任务照常产出，依赖失败项的下游挂起（⊘），goal 标"部分完成"。
3. **升级裁决时**：③对话流 @用户 + 失败节点亮出干预按钮：
   - **重试**（同角色再跑）/ **改派角色**（换 agent）/ **编辑 spec**（改需求文档这条再跑）/ **人工接管** / **跳过**
4. **人工接管形态（已定）**：在**③对话流直接跟该角色 agent 对话，手把手带它**（doctrine"人类介入施工"）。

```
④ 失败时:
   ◆ Goal 重构登录模块              ▓▓▓░ 2/4 · ⚠ 1失败
   ├ ✓ 子任务1 后端API
   ├ ✗ 子任务2 前端登录页  ✗失败:lint error (已自动重试2轮)
   │     [重试][改派][编辑spec][接管][跳过]   ← 升级后亮出
   └ ⊘ 子任务3 测试 (阻塞·依赖2)
```

## 六、本设计与已定 design.md 的衔接

- PMO = squad leader（复用 squad，不新造）。
- goal_run（design.md 2.3）= squad 上挂的一次 goal 派发；子任务 DAG 用 goal_subtask 行 + `depends_on UUID[]` 表达（migration 112 已落地）。
- 状态区数据源 = 既有 WS task:* 事件 + squad member status；复用 ExecutionLogSection/TaskTranscript 思路。
- 失败处理：自动重试 1-2 轮 → 升级裁决；继续其余 + 阻塞下游（树节点标红/⊘），goal 标 partial。
