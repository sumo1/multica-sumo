# goal 模式的触发方式

## 背景

需求二（PMO 复杂任务并行派发）依赖底层 CLI 的「goal 模式」。写需求时这是最大的可行性疑虑——担心要从零造编排引擎。

## 结论

Codex 与 Claude Code 都原生支持 goal 模式，触发方式两种：

- **命令触发**：
  - Codex: `/Goal`（大写 G）
  - Claude Code: `/goal`（小写 g）
- **自然语言触发**：形如「通过目标模式，完成 xxxx」

## 这意味着什么

PMO 管理者不需要自研编排引擎，本质是「把用户的复杂 goal 翻译成对应 runtime 的 goal 触发指令，交给底层 CLI 统筹执行/监控/协调/处理」。

## 仍未证明（design 阶段要查的）

触发方式已知 ≠ 本仓集成已通。仍需读 `server/internal/daemon/` 确认：

1. daemon 当前如何把一条 goal 指令喂给它拉起的 codex / claude-code 进程？
2. goal 模式下的子任务进度，如何回流成既有的 `task:*` WS 事件？
3. 这条链路是否已通、缺什么。
