# 任务模式：独立「任务」页 + PMO 规划层 + 动态小队（实机验证通过）

## 背景

实机 dogfood 后用户反馈三个偏差，2026-06-10 重构为任务模式（design-task-mode.md）：
1. 目标模式不该寄生助理页 → 独立「任务」顶级页。
2. 不该选预建固定 squad → PMO 规划层 + 一个个挑成员，确认时动态建小队。
3. 布局要四栏（讨论+成员+状态树 / 主子会话输出）。

## 核心架构（后端执行引擎零改动）

- **PMO = workspace 默认规划 agent**（`workspace.default_planner_agent_id`，迁移 115）；未设则回退到 workspace 首个可用 agent。讨论主会话 = 跟 PMO 的 chat session（`chat_session.goal_run_id` 关联）。
- **动态小队**：创建任务时自动建「XXX 目标小队」(leader=PMO + 选的成员)，goal 绑这个 squad → 现有调度/对抗验证/干预/接管引擎全部照常跑，零改动。squad 概念保留但创建时机变了（确认/创建时动态建，非用户预建）。
- **三阶段**：CreateTask(discussion) → 多轮和 PMO 讨论 → ConfirmTask(→planning，派 PMO 规划) → 执行引擎接管。

## 关键实现点

- 服务层新增 `CreateTask`/`AddTaskMember`/`ConfirmTask`/`resolvePlannerAgent`（goal.go），独立于旧 `CreateGoal`/`StartPlanning`（thin-slice/显式路仍用）。
- `goal_run.squad_id` 仍 NOT NULL → 选择"创建任务时就建动态 squad"而非让 squad_id 可空，最简。
- ListWorkspaces 显式列表 query 要补 `default_planner_agent_id`，否则 `ListWorkspacesRow` 与 `Workspace` 类型不匹配编译失败（加列副作用）。
- 端点 `/api/tasks`（GET list / POST create / PUT planner）+ `/api/tasks/{id}/members|confirm`。
- 前端：独立 TasksPage 四栏（任务列表 / 讨论 chat + 成员勾选 + GoalStatusTree / 主子会话输出）；删掉助理页的目标模式开关；nav 加「任务」入口（在 用量 和 助理 之间）；paths + reserved_slugs 加 tasks（+ 补了 assistant）。

## 实机验证（真 LLM，2026-06-10）

API 驱动完整跑通：建任务(discussion) → 动态小队「给配置项起名 目标小队」(leader=E2E Agent, member=Hello Bot) + 讨论 chat 绑 PMO → 确认 → planning → PMO 真 claude 拆解 → **3 节点链式工作流,自主插对抗验证**：
```
1. [execute] 提出候选命名方案    → Hello Bot
2. [verify]  对候选命名对抗评审   → E2E Agent  (不同角色!)
3. [execute] 给出最终命名推荐    → Hello Bot
goal → completed
```
computer-use 驱动桌面客户端验证：「任务」nav 入口存在且独立，点击后 TasksPage 渲染（AX 树确认 新建任务/成员/开始讨论 markers）。

## 踩的坑：computer-use 多 tab/window 定位

桌面是 per-tab memory router。computer-use 点 sidebar nav 时，AX element id 编码的是树路径，UI 重渲染后 id 漂移——必须"读完立刻点"，中间隔了快照就点错（点到上一快照的 Skills tab）。且 screenshot 抓的是 frontmost window，AX 树可能返回另一个 window 的内容，两者会不一致。教训：AX 驱动要 fresh-read-then-click，或用 keyword 绑定（升级版 CLI 已支持 `click --keyword`）。
